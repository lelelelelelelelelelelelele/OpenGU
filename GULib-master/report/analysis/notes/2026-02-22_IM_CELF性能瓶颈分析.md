# IM Strategy CELF 性能瓶颈分析

**日期**: 2026-02-22
**问题**: IM 选点在 Cora 上实际耗时 ~515 秒，远超预估的 ~5 秒 (Numba)

---

## 1. 现象

| 来源 | Cora (k=135) 预估耗时 |
|------|----------------------|
| `experiment_params.md` (Numba) | ~5 sec |
| `experiment_params.md` (纯 Python) | ~2 min |
| **实际测量** (`selection_cache/f8bb5e1e745afbb04088032a0dfafc70.json`) | **514.93 sec (~8.6 min)** |

实际耗时是 Numba 预估的 **100 倍**，甚至比纯 Python 预估（~2 min）还慢 4 倍。

## 2. 根因分析

### 2.1 预估只覆盖了 CELF Step 1

`experiment_params.md` 中的预估仅针对 **Step 1（初始 marginal gain 计算）**：

- Step 1: 对 ~1,700 个候选节点各算一次单节点 spread
- 计算量: 1,700 × 100 MC = 170,000 次模拟
- Numba 下确实约 5 秒完成

但完整的 CELF 还有 **Step 2（贪心循环）**，选出 k=135 个节点，这部分被忽略了。

### 2.2 CELF Lazy Evaluation 在低传播概率下退化

CELF 的核心优化是 lazy evaluation：利用子模性，如果某节点的旧 marginal gain 仍是当前最大，就无需重新计算。这依赖于**节点间 gain 差异大**。

当前参数:
- `propagation_prob = 0.1`
- Cora 平均度 ≈ 3.9

在此条件下：
- 单节点 spread ≈ 1.0 ~ 1.3（几乎传不出去）
- **所有候选节点的 marginal gain 非常接近**
- Lazy pruning 几乎不生效，CELF 退化为暴力枚举

退化后的计算量:
```
Step 2 最坏情况: 135 轮 × ~1,700 候选 = ~229,500 次 spread 评估
每次 spread 评估: 100 MC 模拟
总计: ~22,950,000 次 MC 模拟（是 Step 1 的 135 倍）
```

### 2.3 Python 层循环开销

Step 2 的贪心循环在 Python 层执行（`im_strategy.py:244-262`），虽然单次 spread 计算由 Numba 加速，但循环本身存在大量 Python 开销：

```python
# im_strategy.py:255-256 — 在内层循环中执行数十万次
seed_arr = np.array(selected_set_list + [node], dtype=np.int32)  # 每次新建 list + array
base_seed = self.random_seed * 10000 + int(np.sum(np.sort(seed_arr))) % 10000  # 每次排序+求和
```

每次迭代的 Python 开销:
1. `selected_set_list + [node]` — list 拼接（O(i) 拷贝）
2. `np.array(...)` — 新建 numpy 数组
3. `np.sort(seed_arr)` — 排序
4. `np.sum(...)` — 求和
5. `heapq` 操作 — Python heap push/pop

在 ~229,500 次迭代中，这些开销累积显著。

## 3. 耗时拆解（估算）

| 阶段 | 调用次数 | Numba 计算 | Python 开销 | 合计 |
|------|---------|-----------|------------|------|
| Step 1 (初始 gain) | 1,700 | ~5 sec | ~0.5 sec | ~5.5 sec |
| Step 2 (贪心循环) | ~229,500 | ~70 sec | ~440 sec | ~510 sec |
| **总计** | | | | **~515 sec** |

> Step 2 的瓶颈不在 Numba 计算本身，而在 Python 循环 + 对象分配开销。

## 4. 对比：Hybrid 策略为什么快

Hybrid 策略（`hybrid_strategy.py`）只调用 `compute_initial_marginal_gains()`，即 **只做 Step 1**，然后用 TracIn 分数融合排序，跳过了整个 CELF Step 2。

实测: Hybrid 在 Cora 上只需 **~15 秒**（含 TracIn 计算），符合预期。

## 5. 可行的优化方向

### 方案 A: 将 CELF 贪心循环下沉到 Numba

将整个 `_compute_im_celf_numba` 的 Step 2 循环用 `@numba.njit` 实现，包括 heap 逻辑。消除 Python 层开销，预计提速 5-10 倍。

- 难度: 中（需要在 Numba 中实现 min-heap）
- 预期效果: 515s → ~50-100s

### 方案 B: 只用 Step 1 排序（top-k initial gain）

跳过 CELF Step 2，直接取 Step 1 中 spread 最大的 k 个节点。

- 难度: 低
- 预期效果: 515s → ~5s
- 代价: 丢失贪心的集合覆盖优化（但 prob=0.1 时 gain 差异本就极小，集合效应几乎可忽略）

### 方案 C: 提高传播概率

将 `propagation_prob` 从 0.1 提高到 0.3-0.5，使 spread 差异增大，CELF 剪枝生效。

- 难度: 低
- 代价: 改变攻击策略的语义

### 方案 D: 限制 Step 2 评估次数

设置 `max_evaluations` 上限，超过后强制用当前最优。

- 难度: 低
- 预期效果: 可控的计算预算

## 6. 结论

`experiment_params.md` 中的 "~5 sec" 预估有误，**仅覆盖 Step 1**。完整 CELF 在 `prob=0.1` 下 lazy evaluation 退化，实际耗时 ~515 秒，其中 ~85% 是 Python 层循环开销而非 Numba 计算。

**建议**: 对于当前研究，优先考虑 **方案 B**（top-k initial gain），因为低传播概率下 CELF Step 2 的边际收益极小，不值得 100 倍的时间成本。这也与 Hybrid 策略的实践一致。同时应修正 `experiment_params.md` 中的预估数据。
