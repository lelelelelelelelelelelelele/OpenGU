# 组会议题：算力瓶颈与数据集规模分析

> 创建于 2026-02-19，供下次组会讨论

---

## 1. 问题背景

当前实验设备为 **RTX 3060 (12GB VRAM)**。攻击 pipeline 中有两个阶段对大规模图存在算力/显存瓶颈：

1. **TracIn 梯度计算**：逐节点 forward+backward，存储梯度矩阵 `G: [N_train, P]`
2. **CELF-IM MC 模拟**：纯 Python BFS，逐候选节点做 100 次 Monte Carlo 传播模拟

GU 方法本身的 unlearn/retrain 开销相对可控（Cora ~10s，PubMed ~60s）。

---

## 2. 各数据集精确显存/时间估算

### 2.1 TracIn 梯度矩阵 G 的显存

以 2-layer GCN (hidden=64) 为基准：`P = (F × 64 + 64) + (64 × C + C)`

| 数据集 | N_train | Features (F) | Classes (C) | 参数量 P | G 矩阵大小 | 3060 可行？ |
|--------|---------|-------------|-------------|---------|-----------|-----------|
| Cora | ~1,700 | 1,433 | 7 | 92,231 | **0.58 GB** | OK |
| Citeseer | ~2,100 | 3,703 | 6 | 237,446 | **1.86 GB** | OK |
| PubMed | ~12,500 | 500 | 3 | 32,259 | **1.50 GB** | OK |
| Chameleon | ~1,400 | 2,325 | 5 | 149,189 | **0.78 GB** | OK |
| Physics | ~22,000 | 8,415 | 5 | 538,949 | **44.2 GB** | **不可行** |
| ogbn-arxiv | ~107,000 | 128 | 40 | 10,856 | **4.3 GB** | 勉强（剩余空间不足） |

> **Physics 的问题不是节点多，而是 feature 维度大 (8,415)**，导致第一层参数量爆炸。

### 2.2 TracIn 逐节点 backward 的时间

每次 backward 在 3060 上约 3-5ms（Cora-scale GCN）。

| 数据集 | N_train (backward 次数) | 估计时间 |
|--------|------------------------|---------|
| Cora | 1,700 | ~8s |
| Citeseer | 2,100 | ~10s |
| PubMed | 12,500 | ~60s |
| Chameleon | 1,400 | ~7s |
| Physics | 22,000 | ~2 min |
| ogbn-arxiv | 107,000 | ~9 min |

> backward 时间不是瓶颈，G 矩阵存储才是。

### 2.3 CELF-IM MC 模拟的时间

当前实现为**纯 Python BFS**。第一轮需对所有 N_train 个候选节点各做 mc_rounds=100 次 BFS。

Cora 实测 CELF k=50 耗时约 **2 分钟**。按边数和候选节点数线性外推：

| 数据集 | N_train | Edges | 相对 Cora 倍率 | 估计时间 |
|--------|---------|-------|---------------|---------|
| Cora | 1,700 | 5,278 | 1× | ~2 min |
| Citeseer | 2,100 | 4,732 | ~1.1× | ~2 min |
| PubMed | 12,500 | 44,338 | ~62× | ~2 h |
| Chameleon | 1,400 | 36,101 | ~5.6× | ~11 min |
| Physics | 22,000 | 247,962 | ~642× | ~21 h |
| ogbn-arxiv | 107,000 | 1,166,243 | ~14,000× | **不可行** |

> **CELF 是纯 CPU 瓶颈**，与 GPU 无关。租更好的 GPU 无法解决此问题。

---

## 3. 决策分析：优化 vs 租服务器

### 3.1 问题分解

| 瓶颈 | 性质 | 租 GPU 能解决？ | 代码优化能解决？ |
|------|------|---------------|---------------|
| TracIn G 矩阵存储 | GPU 显存 | 是（24GB+ 够 Physics） | 是（mini-batch / 降维） |
| TracIn backward 遍历 | GPU 计算 | 微弱提升 | 不需要（已经很快） |
| CELF MC 模拟 | CPU 计算 | 否（CPU 不是云 GPU 的优势） | **是（必须优化）** |
| GU 方法 unlearn/retrain | GPU 计算 | 是 | 不需要 |

### 3.2 结论：两者都需要，但优先级不同

#### 必须做的代码优化（无论租不租服务器）

1. **CELF 加速**：当前纯 Python BFS 在中等图上就要数小时
   - 方案 A：用 `numba` JIT 编译 BFS 循环，预计 10-50× 加速
   - 方案 B：将 MC 模拟改为批量矩阵运算（稀疏矩阵幂），可 GPU 并行
   - 方案 C：降低 mc_rounds（100→20）或只对 degree top-30% 候选节点做 CELF
   - **推荐**：方案 A + C 组合，改动最小

2. **TracIn 对 Physics 的适配**（如果要跑 Physics）：
   - 方案 A：feature 降维预处理（PCA 到 256 维），P 大幅下降
   - 方案 B：mini-batch 梯度计算，不一次存全部 G 矩阵
   - 方案 C：只对 degree top-20% 候选节点计算 TracIn，其余赋 0
   - **推荐**：方案 A（最干净，不改变方法语义）

#### 租服务器的收益

| 配置 | 价格参考 (AutoDL) | 能解决的问题 |
|------|-------------------|------------|
| A100 40GB | ~3 元/h | Physics TracIn G 矩阵存下（44GB 仍不够，需 80GB 版本或 mini-batch） |
| A100 80GB | ~5 元/h | Physics TracIn 直接存下；大图 GU 方法不 OOM |
| 4090 24GB | ~2 元/h | ogbn-arxiv 以下全部 OK；Physics TracIn 仍需降维 |

### 3.3 推荐方案

```
方案一（省钱，推荐）：本地 3060 + 代码优化
  - 数据集：Cora, Citeseer, PubMed, Chameleon (4 个)
  - 优化 CELF（numba 加速）
  - 不跑 Physics 和 ogbn-arxiv
  - 论文覆盖：3 引文网络 + 1 异质图，中等说服力

方案二（补强，推荐）：本地 3060 + 租 4090 24GB
  - 本地跑：Cora, Citeseer, Chameleon（调试 + 小图）
  - 租机器跑：PubMed, Physics（中大图，需要 CELF 优化后）
  - 优化 CELF（必须）+ Physics TracIn 降维（必须）
  - 论文覆盖：3 引文 + 1 Co-author + 1 异质 = 5 个，说服力好

方案三（最全）：租 A100 80GB
  - 数据集：Cora, Citeseer, PubMed, Physics, Chameleon, ogbn-arxiv (6 个)
  - 仍需优化 CELF（CPU 瓶颈与 GPU 无关）
  - 费用较高，但 ogbn-arxiv 的 scalability 实验对论文加分大
```

## 4. 需要组会讨论确认的问题

1. **数据集范围**：4 个（省时）vs 5 个（方案二）vs 6 个（最全）？投入产出如何取舍？
2. **服务器预算**：是否有实验室经费可用？预估总实验量 ~600 runs（6 策略 × 5 方法 × 4 数据集 × 5 seeds），大图上每 run 含 unlearn + retrain
3. **CELF 优化优先级**：是否先投入时间做 numba 加速（1-2 天工作量），再决定数据集范围？
4. **Physics vs Computers 替代**：Computers (13,752 nodes, 245,778 edges, 767 features) 与 Physics 类似的 co-purchasing 网络但 feature 维度低得多（767 vs 8,415），TracIn G 矩阵仅 ~2.6 GB，是否可以替代 Physics 以规避显存问题？
5. **ogbn-arxiv 的必要性**：这是唯一的 "large-scale" 数据集，reviewer 可能会问 scalability。但代价是 CELF 必须重写 + 需要大显存。值得吗？

## 5. 附录：新增指标 (v2) 对算力的额外影响

| 指标 | 额外开销 | 说明 |
|------|---------|------|
| F1 Drop | 无 | 两次 forward，忽略不计 |
| MIA AUC | 无 | 一次 forward |
| Approximation Gap | **每组实验多一次 retrain** | 时间翻倍，但不增加显存峰值 |
| Collateral Damage | 无 | 两次 forward |
| Gap 统计显著性 | **每个配置跑 ≥5 seeds** | 总实验量 ×5 |

核心影响：retrain 和多 seed 导致**总实验时间 ≈ 单次 × 10**（retrain ×2, seeds ×5）。这进一步说明 CELF 加速是必做项——如果单次 CELF 要 2 小时，那 5 seeds 就是 10 小时。
