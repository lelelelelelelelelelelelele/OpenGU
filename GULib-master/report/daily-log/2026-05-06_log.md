### 临结束 — 用户的反思

> "这玩意又不是 deterministic 的。不知道有多少造假的。"

我给的回答：GPU 浮点 reduction 永远做不到 bit-identical（cudnn atomicAdd / numba 多线程合并），但科研可信度 ≠ bit-identical。可信度 = **属性级 reproducibility**：
- 同 seed N 次跑 mean ± CI 重合 ✅
- 改 hyperparam 应改的方向 ✅
- 不改 hyperparam 结果应不变 ✅

8 个 bug 跟噪声本质不同——是**系统性偏差**（cache 错了所有 hybrid α 都是同一份选点，100% 偏差，不是 noise）。修复后剩下的是真噪声，5 seed 平均 + CI 后还在的就是 signal。

提议明天写 `tests/test_phase_b_invariants.py` 5 条属性测试 lock-in 这 8 个 bug 不复现，**不依赖 bit-identical**：
1. Hybrid α=0.3 选点 ≠ α=0.7 选点（cache key 修复）
2. random/degree/pagerank ⊂ train_mask（baseline 修复）
3. TracIn 在 trained model 上选点 ≠ random-init 上选点（random-init 修复）
4. compare_strategies 顺序无关性（训练隔离）
5. config.unlearning_path 与 inject path 一致（路径一致性）

5 条属性都是离散是非题，比"看 f1_after 数字"靠谱。

## 💡 学到的事

1. **Reviewer 模式比 Builder 模式找 bug 多**：codex 第一轮跑 4 个 fix；我做 reviewer 后第二轮 audit 又找出 3 个 codex 没碰过的角落（SelectionCache / ResultCache / 训练状态污染）。两种角色互补——builder 修可见的 symptom，reviewer 找潜在的不变量违反。下次大改前应该 build → review → build → review 多轮。

2. **Cache key 完整度是隐形 correctness invariant**：`_strategy_params_for_cache` 对 hybrid 返回 `{}` 这条 bug 存在了不知道多少 commit。表面上 Hybrid 能跑、能写文件、能进 cache，但 cache key 错了导致两次实验互相覆盖——**功能性正确不等于 cache 正确**。每次新增 hyperparam 都要问"它进 cache key 了吗"。

3. **训练状态污染比 random-init 还隐蔽**：random-init bug 至少 strategy 之间相对一致（都用 random init）；训练状态污染让 strategy 顺序敏感——同 yaml 调换 strategies 列表顺序数字就变。下次任何"共享 mutable state across iterations"的设计都要警惕。

4. **bit-identical 是奢望，property-based reproducibility 是底线**：用户最后那句"不知道有多少造假"反映的不是 deterministic 焦虑，是对系统性 bug 的信心崩溃。回答应该是"区分系统性 bug（100% 偏差）vs 浮点噪声（5 seed CI 内）"，而不是"我们已经 deterministic 了"。`test_phase_b_invariants.py` 测属性而非数值是这个思路的 codification。

5. **用户清盘的决策力**：发现 cache_schema sentinel 后，正确的做法不是堆代码兼容旧条目，而是**rm 掉物理证据，让代码干净**。`你别升级v2.好好改` 这句指令明确把 deprecation 责任从 sentinel 转移到 git history（commit message 自带 v1 vs v2 上下文）。代码里少一个 dead branch 比 dashboard 多一行 changelog 重要。

6. **Daily log 的"事实流" vs "决策流"**：今天的工作不只是 5 个 commit + 8 个 bug 数字，更多是**两轮 audit 之间的认知升级**——从 "TracIn 测的是 IF" → "测的是 random-init 代理" → "cache 误命中所以根本没在测" → "训练状态还在污染"。Dashboard 的 V-entry 只记 fact，daily log 记 decision flow。两者互补。
